# Concerns

## Tech Debt

### High Priority

**Stub tasks in `src/workers/tasks.py`**
- `process_transcript` and `generate_brief` return `{"status": "queued"}` placeholders
- They do no actual work â€” tests pass because they only validate arg presence
- Risk: the test suite gives false confidence; real pipeline uses `ingest.py` / `extract.py` instead
- File: `src/workers/tasks.py`

**`/calendar/today` endpoint is a stub**
- Returns `{"upcoming_meetings": []}` unconditionally
- Google Calendar integration not yet implemented
- File: `src/api/routes/calendar.py`

**`/conversations/{id}/connections` is a stub**
- Returns an empty list; Connection graph not yet built
- File: `src/api/routes/conversations.py`

**Brief generation not implemented**
- `src/models/brief.py` exists but no route or worker creates briefs
- The `brief_*` junction tables exist in DB but are unpopulated

### Medium Priority

**No pagination on list endpoints**
- `/conversations`, `/topics`, `/commitments` return all records
- Could become slow as user data grows
- Files: `src/api/routes/conversations.py`, `topics.py`, `commitments.py`

**Ingest pipeline fires extract + embed as fire-and-forget `.delay()` calls**
- No error handling if downstream tasks fail after `ingest_recording` succeeds
- Conversation exists in DB but may have no embeddings or extracted entities
- File: `src/workers/ingest.py` lines 308â€“317

**`get_direct_connection()` bypasses RLS**
- Used only in `/search` with explicit `WHERE user_id = %s`
- Requires careful review of every future query that uses this path
- File: `src/database.py`

## Security

**`service_role` key usage scope**
- `get_admin_client()` exists and is well-documented as forbidden from routes/workers
- But there's no static analysis enforcement â€” a developer could call it from a route
- Recommendation: Add a mypy plugin or grep CI check to detect `get_admin_client` outside migration scripts

**Google tokens stored in DB**
- `user_index.google_access_token` and `google_refresh_token` stored as plain text columns
- These are sensitive credentials â€” no encryption at rest beyond Supabase's default
- Migration `003_google_tokens.sql` adds these columns

**No rate limiting**
- No rate limiting on `/onboarding` bulk import endpoint
- A user could trigger many concurrent `ingest_recording` tasks
- Celery provides some throttling via `soft_time_limit` per task but no per-user limits

## Performance

**pgvector search uses raw psycopg2**
- Necessary because Supabase PostgREST doesn't support `<=>` operator
- Each search opens and closes a new connection â€” no connection pooling
- File: `src/api/routes/search.py`, `src/database.py:get_direct_connection`

**Transcript parsing is synchronous in Celery task**
- `_parse_google_transcript` runs inline â€” fine for most transcripts
- Very long transcripts (3hr+ meetings) could hit `soft_time_limit=300`
- File: `src/workers/ingest.py`

**OpenAI embedding: segments embedded individually or in batch?**
- `embed_conversation` task exists but batching behavior depends on implementation
- Check `src/workers/embed.py` for batch size â€” large conversations may be slow

## Fragile Areas

**Transcript format detection relies on timestamp regex**
- `_TS_DETECT_RE = re.compile(r"^\d{1,2}:\d{2}(:\d{2})?$", re.MULTILINE)`
- If Google Meet changes export format, parser silently falls back to Gemini Notes parser
- File: `src/workers/ingest.py:_detect_and_parse`

**No retry on Google token refresh**
- `refresh_access_token_sync` is called without retry logic
- If Google's OAuth server is slow, the entire ingest task fails and retries from scratch (2 max retries, 60s delay)
- File: `src/workers/ingest.py`

**`drive_file_id` uniqueness constraint**
- `UNIQUE INDEX (user_id, drive_file_id) WHERE NOT NULL` â€” good for idempotency
- But the idempotency check is done manually before insert (TOCTOU window)
- A race condition could cause two simultaneous ingests to both pass the check before the first commits
- Low risk given Celery serialization but worth noting

## Missing Features / Test Coverage Gaps

**No end-to-end test for full pipeline**
- Unit tests cover individual tasks in isolation
- No test covers `ingest â†’ extract â†’ embed` chain
- RLS isolation tests cover DB layer but not full API â†’ worker flow
